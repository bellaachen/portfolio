讓交易 App 真正可以交付 Turning a Design into

Turning a Design into a Shippable Product Taking a trading app from Happy-Path mockups to a fully specified, released product.

產品Product
證券交易 AppSecurities Trading App
期間Timeline
2025 — 2026
專案背景Background

一個可以 Demo，卻很難交付的產品A Product That Could Demo, but Couldn't Ship

2025 年初，我開始接手公司的交易 App。這個 App 並不是從零開始 — 外包團隊已經完成第一版產品，也完成了大部分功能。 In early 2025, I took over the company's trading app. It wasn't a greenfield project — an outsourced team had already delivered a first version with most features in place.

然而，當團隊準備接手持續開發時，很快就發現，產品雖然可以 Demo，卻很難真正交付。很多流程缺乏完整定義，使用體驗不一致，部分畫面與 Backend API 不符，許多功能只有 Happy Path，缺少實際開發所需要的各種產品規則。 But as our team prepared to take over ongoing development, we quickly realized the product could be demoed but was very hard to actually ship. Many flows were incompletely defined. The experience was inconsistent. Some screens didn't match the backend APIs. Most features only covered the Happy Path, missing the product rules real development requires.

因此，產品團隊決定重新設計整體介面。設計師重新繪製了一版新的 App — 但仍然只有 Happy Path。 The product team decided to redesign the entire interface. Our designer produced a fresh version of the app — but it, too, only covered the Happy Path.

而我接手的工作，就是把這份設計稿，變成真正可以被工程實作、測試與上線的產品。 My job was to turn that design file into a product that could actually be built, tested, and released.
Challenge

設計稿回答不了工程師的問題The Design Couldn't Answer the Engineers' Questions

我很快發現，真正困難的並不是新增功能，而是設計稿回答不了工程師真正需要知道的問題。例如： I quickly found that the hard part wasn't adding features — it was that the design couldn't answer the questions engineers actually needed answered. For example:

  • API 回傳不同資料時，畫面要怎麼顯示？How should the screen render when the API returns different data?
  • Loading、Error、Empty State 怎麼處理？How are Loading, Error, and Empty states handled?
  • 沒有權限時要看到什麼？What does a user without permission see?
  • 哪些狀態需要提示使用者？Which states need to notify the user?
  • 同一個功能在不同市場是否有不同規則？Does the same feature follow different rules in different markets?

如果沒有這些定義，每位工程師都可能做出不同版本。 Without these definitions, every engineer could end up building a different version.

我的工作My Role

不是從功能開始，而是先建立產品規格Not Features First — Specifications First

因此，我沒有從功能開始，而是先建立產品規格。我負責把 Happy Path 補齊成完整的 Product Specification。 So I didn't start with features. I started by building the product specification — expanding the Happy Path into a complete Product Specification.

01

補齊產品規格Completing the Specification

與設計師一起完成所有 Edge Cases、Loading、Error Handling、Display Rules 與例外流程，讓每個畫面都有明確規則。Worked with the designer to define every edge case, loading state, error handling rule, display rule, and exception flow — so every screen had explicit rules.

02

對齊 UI 與 BackendAligning UI with Backend

重新確認每個畫面的 API Mapping。與 Backend 一起確認資料來源，再重新調整畫面顯示方式，而不是要求工程修改 API 配合設計。Re-verified the API mapping for every screen. Confirmed data sources with the backend team, then adjusted how screens displayed data — rather than asking engineering to change APIs to fit the design.

03

建立一致的產品規則Building Consistent Product Structure

隨著功能增加，我重新整理既有 Feature 的分類方式，讓後續需求有一致的產品結構，也降低跨團隊溝通成本。As features grew, I reorganized how existing features were categorized, giving later requirements a consistent product structure and lowering cross-team communication cost.

04

推動產品上線Driving the Release

參與整個 Release Planning：Apple App Store Review、不同市場版本策略、Release 時程安排與跨團隊協調。Took part in the full release planning: Apple App Store review, per-market version strategy, release scheduling, and cross-team coordination.

我的工作，不只是完成需求。更重要的是，讓產品真正可以交付。 My job wasn't just to complete requirements — it was to make the product genuinely shippable.
代表性產品決策Key Product Decisions

三個決策，三種取捨Three Decisions, Three Trade-offs

當 Backend 支援即時更新後，優先改善核心交易體驗When the Backend Supported Real-time Updates, Core Trading Experience Came First

原本 Equity、P&L 等帳戶資訊每十五分鐘才更新一次。當 Backend 已支援即時更新後，我調整產品優先順序，優先推動帳戶資訊即時刷新。Account information like Equity and P&L originally refreshed only every fifteen minutes. Once the backend supported real-time updates, I reprioritized the roadmap to bring real-time account data first.

為什麼：Why: 對交易產品而言，資訊是否可信，比增加更多功能更重要。For a trading product, whether information can be trusted matters more than adding features.

同一個產品，兩種登入策略One Product, Two Login Strategies

由於歷史架構限制，全球版本仍使用 Web Login；中國版本則改採 Native Login，並加入生物辨識登入。我負責定義兩套登入流程、畫面規則、Edge Cases 與產品 Scope，讓不同市場都能維持一致的使用體驗。Due to legacy architecture constraints, the global version kept Web Login, while the China version moved to Native Login with biometric authentication. I defined both login flows — screens, rules, edge cases, and scope — so each market kept a consistent experience.

為什麼：Why: 市場條件不同時，一致的「體驗」比一致的「實作」更重要。When market conditions differ, a consistent experience matters more than a consistent implementation.

用 Release Strategy 解決 App Store 上架問題Solving App Store Approval with Release Strategy

金融產品上架除了功能完成之外，還需要符合不同市場的法規與 Apple Review 要求。我參與規劃版本策略與 Release 時程，協助團隊降低 App Review 帶來的不確定性，讓產品能順利完成 Rollout。Shipping a financial product isn't just about finishing features — it must satisfy market regulations and Apple's review requirements. I helped plan the version strategy and release schedule, reducing the uncertainty of App Review so the product could complete its rollout.

為什麼：Why: 上架審核是產品問題，不只是流程問題 — 它需要被設計，而不是被等待。App review is a product problem, not just a process problem — it needs to be designed for, not waited on.
結果Outcome

第一次擁有一套完整且一致的產品規格The Product's First Complete, Consistent Specification

經過一年多的迭代，產品正式完成 Rollout。 After more than a year of iteration, the product completed its full rollout.

回頭來看，我最大的貢獻並不是新增了多少功能，而是讓產品第一次擁有一套完整且一致的產品規格 — 讓設計、工程與 Backend 都能依照相同的規則協作，也讓後續功能能建立在一致的基礎上持續演進。 Looking back, my biggest contribution wasn't the number of features added. It was giving the product its first complete and consistent specification — one that let design, engineering, and backend collaborate by the same rules, and let every later feature evolve on a consistent foundation.

Reflection

PM 的價值，是建立規則A PM's Value Is Building the Rules

這個專案讓我重新理解 Product Manager 的價值。一開始，我以為 PM 的工作是規劃功能。但真正參與產品落地後，我發現更重要的是建立規則。 This project changed how I understand a Product Manager's value. I used to think a PM's job was planning features. After carrying a product all the way to launch, I found what matters more is building the rules.

設計可以畫出理想流程。Backend 可以提供資料。工程可以完成開發。而 PM 的工作，是把這些不同角色的觀點整合成一套所有人都能共同理解的產品規格。 Design can draw the ideal flow. Backend can provide the data. Engineering can build it. The PM's job is to integrate those perspectives into one product specification everyone can understand together.

也是從這個專案開始，我養成了一個習慣。每接手一個功能，我不會先問「這個功能怎麼做？」，而是先問： This project is also where I formed a habit. Whenever I take on a feature, I no longer start by asking "how do we build this?" — I start by asking:

接手任何功能前，我會先確認Before any feature, I first confirm
  • 設計是否真的可以被開發？Can this design actually be built?
  • API 是否支援？Do the APIs support it?
  • 所有狀態都有定義嗎？Is every state defined?
  • 是否考慮了所有例外情境？Have all the exception scenarios been considered?

因為我相信，一個成熟的產品，不是因為功能很多，而是因為每一次迭代，都建立在一套清楚且一致的產品規則之上。 Because I believe a mature product isn't one with the most features — it's one where every iteration is built on a clear, consistent set of product rules.
